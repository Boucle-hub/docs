The Ruffus API lets you read and write your loyalty data ‚Äî locations, cards, customers, balances, rewards, and event feeds ‚Äî from your own systems. It is a server-to-server API: requests from browsers are unsupported by design (no CORS headers are sent), because API keys must never ship in front-end code.

## Base URL

```text
https://api.ruffus.app/v1
```

Only `/v1/*` paths are served on this host. All responses are JSON.

## Authentication

Create keys from **Dashboard ‚Üí API keys** (owner role required). The full key ‚Äî format `rk_live_‚Ä¶` ‚Äî is shown **once** at creation; store it securely. Only a hash is kept on our side, so a lost key can't be recovered, only revoked and replaced.

Pass the key on every request, either way:

```bash
curl https://api.ruffus.app/v1/ping \
  -H "X-Api-Key: rk_live_..."

# or
curl https://api.ruffus.app/v1/ping \
  -H "Authorization: Bearer rk_live_..."
```

`GET /v1/ping` is the connectivity check ‚Äî it echoes your merchant id, the key's scopes, and your plan tier, and requires no particular scope.

`rk_test_` keys are reserved for a future sandbox mode and are currently rejected with `test_mode_unsupported`.

## Scopes

Each key carries an explicit set of scopes, chosen at creation:

| Scope | Grants |
| --- | --- |
| `customers:read` | Customer lookup and membership balances |
| `customers:write` | Enrolling existing customers at a location |
| `loyalty:read` | Locations, cards, rewards, and event feeds |
| `loyalty:write` | Issuing stamps, adjustments, and redeeming rewards |
| `webhooks:manage` | Webhook endpoint management |

## Plan entitlements

| Plan | API access |
| --- | --- |
| Basic | Not available |
| Growth | Read scopes \+ `webhooks:manage`, 300 requests/min per key |
| Network | All scopes, 600 requests/min per key |
| Custom | All scopes, 600 requests/min per key |

If your plan changes, existing keys are kept: requests using scopes your new plan lacks return `plan_not_entitled` until you upgrade again.

## Rate limits

Per-key limits are listed above; an additional per-account ceiling applies across all keys. When you exceed a limit the response is `429` with `Retry-After`, `X-RateLimit-Limit`, `X-RateLimit-Remaining`, and `X-RateLimit-Reset` headers.

## Errors

Every error uses one envelope:

```json
{
  "error": {
    "code": "missing_scope",
    "message": "This key does not have the 'loyalty:read' scope.",
    "requestId": "req_5f0c‚Ä¶"
  }
}
```

`code` is stable and machine-readable; `message` is human-readable and may change. `requestId` is echoed in the `X-Request-Id` header on every response (success included) ‚Äî quote it when contacting support.

| Code | Status | Meaning |
| --- | --- | --- |
| `invalid_api_key` | 401 | Missing, malformed, or unknown key |
| `expired_api_key` | 401 | Key past its expiry date |
| `revoked_api_key` | 401 | Key was revoked |
| `test_mode_unsupported` | 403 | `rk_test_` keys not yet supported |
| `missing_scope` | 403 | Key lacks the required scope |
| `plan_not_entitled` | 403 | Your plan doesn't include this capability |
| `rate_limited` | 429 | Too many requests |
| `validation_error` | 400 | Bad parameters ‚Äî see `details[]` per field |
| `not_found` | 404 | No such resource in your account |
| `conflict` | 409 | The request can't be applied in the resource's current state ‚Äî see the message |
| `subscription_inactive` | 409 | The location's Ruffus subscription isn't active |
| `idempotency_conflict` | 422 | `Idempotency-Key` reused with a different request body |
| `internal_error` | 500 | Something went wrong on our side |

## Idempotency

Write endpoints accept an optional `Idempotency-Key` header (1‚Äì255 chars) ‚Äî reuse the same key to safely retry a request (network timeout, no response received) without double-applying it.

- First request with a key: processed normally, and the response is stored against that key for **24 hours**.
- Retried with the **same** key and the same body: the stored response is replayed verbatim, byte-for-byte, with an `Idempotent-Replay: true` header ‚Äî safe to retry as many times as you like.
- Retried with the **same** key and a **different** body: rejected with `422 idempotency_conflict`.
- Concurrent requests using the same not-yet-finished key: the second one gets `409 conflict` ‚Äî wait for the first to finish, then retry with the same key if needed.
- A request that fails with a `5xx` releases its key immediately, so you can retry right away; `4xx` responses are stored and replayed like any other outcome.

Keys are scoped to the API key that made the request ‚Äî reusing a key value across two different API keys does not collide. This is separate from the `external_id` field on the [`/earn` and `/adjust`](/api/write-endpoints) endpoints, which deduplicates by *business meaning* (e.g. "this POS transaction") rather than by *request delivery* ‚Äî use `Idempotency-Key` for retry-safety and `external_id` for your own dedup key, together if you like.

## Pagination

List endpoints are cursor-paginated:

- `limit` ‚Äî 1 to 100, default 25
- `starting_after` ‚Äî the `id` of the last item from your previous page

```json
{ "object": "list", "data": [ ‚Ä¶ ], "hasMore": true }
```

When `hasMore` is `true`, request the next page with `starting_after` set to the last item's `id`. Ordering is stable (creation time, then id).

## Dates and privacy

All timestamps are UTC ISO-8601 with a `Z` suffix. Customers who exercised their right to erasure return `not_found` on lookup and appear in event feeds with `"customerId": null, "anonymized": true`. Email and phone appear only on the direct customer endpoint, never in lists or feeds.
